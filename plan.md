# Binaried implementation plan

## Purpose

Build a small, polished task manager with an Angular standalone frontend, an Express REST backend, and MongoDB persistence. Bun is the only package installer. The repository uses a Bun workspace with separate `client` and `server` packages.

## Chosen product decisions

- Authentication is deliberately dummy-only. The backend accepts `admin` / `admin123`; the client stores an `isLoggedIn` flag in local storage. There are no tokens, sessions, JWT middleware, JWT secrets, or HTTP token interceptors.
- `/register` is an explanatory page, not an account-creation flow. It makes the demo limitation clear and returns people to login.
- MongoDB runs through Docker Compose for local development.
- Tasks have a required date (not a time), and overdue dates remain valid.
- Tailwind provides the responsive, pastel visual system. Angular CDK is used only for the accessible deletion confirmation dialog.

## Domain model

### Demo User

The sole v1 identity is `admin`. A backend middleware attaches this fixed identity to task requests. The task API never accepts an owner identifier from the client.

### Task

| Field | Rules |
| --- | --- |
| `title` | Required, trimmed, 1вЂ“120 characters |
| `description` | Optional, trimmed, up to 1,000 characters |
| `status` | `todo`, `in-progress`, or `done`; defaults to `todo` |
| `dueDate` | Required calendar date, stored as a MongoDB `Date` |
| `ownerId` | Internal fixed value: `admin` |
| `createdAt` | Generated by MongoDB/Mongoose |

## Public HTTP interface

- `GET /api/health`
- `POST /api/login`
- `GET /api/tasks?status=all|todo|in-progress|done&sort=dueDate|createdAt|status&direction=asc|desc`
- `POST /api/tasks`
- `GET /api/tasks/:id`
- `PATCH /api/tasks/:id`
- `DELETE /api/tasks/:id`

Errors return a stable JSON `message`, with field-level details for invalid task input. The server never sends stack traces to the browser.

The Task module owns task validation, ownership, filtering, and persistence. Controllers keep a narrow interface to HTTP concerns. A Mongoose repository wrapper is intentionally deferred: with only one persistence adapter it would be a shallow module and would not improve locality.

## Local setup and commands

1. Install Bun, restart zsh, and confirm `bun --version`.
2. Run `bun install` from the repository root after the workspace manifests exist.
3. Copy `server/.env.example` to `server/.env`.
4. Start MongoDB with `docker compose up -d mongo`.
5. Start the server and client using the root workspace scripts in separate terminals, or the combined `bun run dev` script.
6. Run backend tests, Angular tests, and the production client build before every reviewed commit.

No secret belongs in Git. Docker volumes, dependency directories, build output, coverage, logs, and local environment files remain ignored.

## Commit phases

1. `docs: add implementation roadmap and repository conventions`
2. `chore: configure Bun workspace and local MongoDB`
3. `feat(server): scaffold Express application foundation`
4. `feat(server): add dummy login and task CRUD endpoints`
5. `test(server): cover auth and task API flows`
6. `feat(client): scaffold standalone Angular workspace`
7. `feat(client): add guarded dummy authentication flow`
8. `feat(client): add task dashboard and form workflows`
9. `style(client): refine responsive pastel task experience`
10. `docs: complete setup guide and delivery notes`

Before every commit: inspect the scoped diff, run `git diff --check`, run that phase's verification, and confirm `git status --short` contains only the intended files. Do not amend, squash, reset, or mix later-phase work into an earlier commit.

## Verification requirements

- Backend integration tests cover login success/failure, task CRUD, invalid requests, missing tasks, owner isolation, filtering, and sorting.
- Angular tests cover guards, HTTP data access, form validation, and route-level behavior.
- Manual browser checks cover login/logout, deep-link guarding, empty/loading/error states, CRUD, dirty-form protection, deletion confirmation, URL-synced filter/sort state, and 320 px through desktop layouts.
- UI review checks visible focus, semantic landmarks, labels, keyboard operation, live error feedback, reduced motion, long-text handling, responsive overflow, and accessible destructive confirmation.

## Deferred work

Real registration, persistent users, bcrypt, JWT or session authentication, pagination, automated browser E2E tests, real-time updates, roles, CI/CD, and deployment are deliberately deferred.
